🌐 Why User & Group Management?

Before diving into commands and files, it's crucial to understand why Linux uses this architecture. Linux is built from the ground up as a multi-user, multi-tasking operating system.

🔒 Security & Isolation

If a malicious program runs under a standard user account, it can only damage that user's files. It cannot destroy the entire system because it lacks root privileges.

🔍 Accountability

When everyone has their own account, system logs can accurately track who did what, and when. Shared accounts destroy accountability.

⚖ Least Privilege

Groups allow administrators to grant exactly the permissions needed for a job without giving out full root access.

The "Everything is a File" Philosophy

In Linux, almost everything (hardware devices, processes, directories) is treated as a file. Access to these files is strictly controlled by ownership (User and Group). Therefore, mastering users and groups means mastering access control for the entire system.

📂 Core Configuration Files

Linux systems rely on standard plain-text files to manage identity, authentication, and authorization.

/etc/passwd

Contains user account information. World-readable but only writable by root. It no longer contains actual password hashes.

root:x:0:0:root:/root:/bin/bash
alice:x:1000:1000:Alice Admin:/home/alice:/bin/bash

Format Breakdown (Separated by :)

  • Username: Login name
  • Password: 'x' indicates the actual hash is stored safely in /etc/shadow
  • UID: User ID (0 for root, 1000+ for regular users)
  • GID: Primary Group ID
  • GECOS: Comments/Full Name
  • Home Directory: Path to the user's home folder
  • Shell: Default command interpreter (e.g., /bin/bash)

/etc/shadow

Contains encrypted passwords and password aging information. Accessible only by root to prevent password cracking.

alice:$6$xyz...:19450:0:90:7:::

Format Breakdown (Separated by :)

  • Username: Login name (must match /etc/passwd)
  • Password Hash: Encrypted password ($6$ indicates SHA-512)
  • Last Change: Days since Jan 1, 1970 that the password was last changed
  • Min Age: Minimum days before the user is allowed to change it again
  • Max Age: Maximum days before the password expires
  • Warning: Days to warn the user before the password expires
  • Inactive: Days after expiration before the account is permanently locked
  • Expire: Absolute date (days since Jan 1, 1970) when the account expires
  • Reserved: Unused field

/etc/group

Defines groups on the system and assigns users to them for shared file access.

developers:x:1005:alice,bob

Format Breakdown (Separated by :)

  • Group Name: Name of the group
  • Password: Usually 'x' (group passwords are rarely used today)
  • GID: Group ID number
  • User List: Comma-separated list of supplementary members

👤 Local User Accounts

Command-line utilities provide a safe, automated way to modify the core configuration files securely without manually editing them.

useradd vs. adduser (RHEL Context)

In many tutorials, you might see adduser. On Debian/Ubuntu systems, adduser is an interactive script that asks for passwords and names step-by-step.

However, on RHEL/CentOS systems, adduser is usually just a symbolic link pointing exactly to useradd. In the RHEL world, you must use useradd with the appropriate flags (like -m) to set up the account properly, and then use passwd to set the password.

useradd - Create a new user

Creates a new user account.

sudo useradd -m -d /opt/jsmith -s /bin/bash -c "Developer" -G docker jsmith
-m : Create the home directory (Crucial flag!)
-d [dir] : Specify a custom home directory path
-s [shell] : Define the login shell (e.g., /bin/bash)
-c [comment] : Add GECOS info (Full name, etc.)
-G [group1,group2] : Add to supplementary groups
-u [UID] : Force a specific User ID
-e [YYYY-MM-DD] : Set an account expiration date

usermod - Modify an existing user

sudo usermod -aG wheel jsmith
sudo usermod -L jsmith
-aG [group] : Append user to a supplementary group
-l [new_name] : Change the user's login name
-L : Lock the user's password
-U : Unlock a locked account
-d [new_dir] -m : Move home dir contents to new location
Critical Warning: Always use -aG together when appending a user to a supplementary group. Using only -G will overwrite and remove their current supplementary groups!

userdel - Delete a user

sudo userdel -r jsmith
-r : Removes the user's home directory and mail spool. Without this, the home directory remains orphaned.
-f : Force deletion, even if the user is currently logged in.

Password Management

passwd

Used to set or change user passwords. Root can change any password.

sudo passwd jsmith

chage - Password aging policies

Manages when a password expires, enforcing security policies.

sudo chage -M 90 -W 7 -E 2026-12-31 jsmith
-M [days] : Max days before password change required
-m [days] : Min days between password changes
-W [days] : Warning days before expiration
-E [YYYY-MM-DD] : Account expiration date
-l [user] : List current aging info

👥 Group Management

Groups are used to organize users and manage permissions for files and directories efficiently.

Primary Group

Every user has exactly one primary group. When a user creates a new file, it is automatically assigned to this primary group. Specified by the 4th field in /etc/passwd.

Supplementary Groups

Users can belong to zero or many supplementary (secondary) groups. These grant additional access permissions to files owned by those groups. Managed in /etc/group.

groupadd

Create a new group.

Options: -g [GID] (Force specific Group ID)

sudo groupadd -g 2000 developers

groupmod

Modify a group.

Options: -n [new_name], -g [new_GID]

sudo groupmod -n devops developers

groupdel

Delete an existing group.

sudo groupdel devops

🔑 Privileged Access: sudo & visudo

Logging in directly as 'root' is dangerous. Best practice dictates using standard accounts and escalating privileges via sudo (Superuser DO) only when necessary.

The /etc/sudoers File

This file controls who can run what commands as whom on which machines. NEVER edit this file directly with vi or nano. Always use visudo.

Why visudo?

visudo locks the sudoers file against multiple simultaneous edits and performs a strict syntax check before saving. If you introduce a syntax error using a normal editor, you will break sudo and lock everyone out of root access!

sudo visudo

Understanding sudoers Syntax

alice ALL=(ALL:ALL) ALL
%wheel ALL=(ALL) NOPASSWD: ALL
  • User/Group: alice (user) or %wheel (group - indicated by %).
  • Host: ALL (First ALL means this applies to all hosts).
  • RunAs: (ALL:ALL) (Can run commands as any user:group).
  • Commands: NOPASSWD: ALL (Can run any command without prompting for a password).

Granting Specific Command Execution

You don't always want to give full root access. You can grant access to only specific commands. Important: Always use absolute paths for commands in sudoers.

Example: Give user bob permission to manage users, without requiring a password, but nothing else.

bob ALL=(ALL) NOPASSWD: /usr/sbin/useradd, /usr/sbin/usermod, /usr/sbin/userdel, /usr/bin/passwd

With this line in sudoers, Bob can run sudo useradd newuser, but if he tries sudo rm -rf / or sudo systemctl restart httpd, he will be denied.

Full Access the Easy Way (RHEL Context)

Instead of modifying the sudoers file, RHEL systems pre-configure the wheel group to have full sudo rights. Simply add a user to this group.

sudo usermod -aG wheel target_username

Scenario 1

Create a new user named 'sarah'. Ensure her home directory is created, her default shell is /bin/bash, and force her UID to be exactly 2050.

sudo useradd -m -s /bin/bash -u 2050 sarah

Scenario 2

A new group called 'audit' needs to be created. Then, append existing user 'sarah' to this group without removing her from any other groups.

sudo groupadd audit
sudo usermod -aG audit sarah

Scenario 3

Set a policy for 'sarah' so her password expires every 60 days, and she gets a warning 10 days before it expires.

sudo chage -M 60 -W 10 sarah

Scenario 4

Grant 'sarah' full privileged access using the standard RHEL group method without using visudo.

sudo usermod -aG wheel sarah

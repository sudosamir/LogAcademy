Trust On First Use (TOFU) & Known Hosts Trust On First Use (TOFU) və Known Hosts

SSH mitigates Man-in-the-Middle attacks by tracking server identity keys. System-wide known hosts are stored in /etc/ssh/ssh_known_hosts, while user-specific host fingerprints are stored in ~/.ssh/known_hosts. SSH, server Identity Key-lərini izləyərək Man-in-the-Middle hücumlarını azaldır. Sistem miqyaslı known hosts faylları /etc/ssh/ssh_known_hosts-da, istifadəçiyə xüsusi host fingerprint-ləri isə ~/.ssh/known_hosts-da saxlanılır.

Troubleshooting: Host Key Changed
# Attempting to log into a server that was recently reinstalled [student@workstation ~]$ ssh root@serverb @@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@ @ WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED! @ @@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@ IT IS POSSIBLE THAT SOMEONE IS DOING SOMETHING NASTY! Someone could be eavesdropping on you right now (man-in-the-middle attack)! It is also possible that a host key has just been changed. Offending ED25519 key in /home/student/.ssh/known_hosts:12 Host key verification failed.
# Fix: Remove the old cached fingerprint for serverb [student@workstation ~]$ ssh-keygen -R serverb # Host serverb found: line 12 /home/student/.ssh/known_hosts updated.
AuthenticationAutentifikasiya

Key-Based Authentication

Passwords are vulnerable to dictionary attacks. Deploying cryptographic key pairs is the enterprise standard for SSH authentication. Parollar dictionary hücumlarına qarşı zəifdir. Kriptoqrafik key pair-lərin tətbiqi SSH autentifikasiyası üçün enterprise standartıdır.

1

Generate the PairPair-i Yarat

RHEL 9 strongly recommends the ed25519 algorithm over legacy RSA. RHEL 9, köhnə RSA yerinə ed25519 alqoritmini qətiyyətlə tövsiyə edir.

[student@ws ~]$ ssh-keygen -t ed25519 Generating public/private ed25519 key pair. Enter file to save key (/home/student/.ssh/id_ed25519): [Enter] Enter passphrase (empty for no passphrase): ****** Your identification has been saved in ...id_ed25519 Your public key has been saved in ...id_ed25519.pub
2

Deploy Public KeyPublic Key-i Deploy Et

ssh-copy-id securely appends your .pub key to the remote system and strictly sets SELinux contexts and permissions. ssh-copy-id təhlükəsiz şəkildə sizin .pub key-inizi remote sistemə əlavə edir və SELinux kontekstlərini və icazələri ciddi şəkildə təyin edir.

[student@ws ~]$ ssh-copy-id student@servera student@servera's password: Number of key(s) added: 1 Now try logging into the machine, with: "ssh 'student@servera'" and check to make sure that only the key(s) you wanted were added.

The File Permission Matrix (CRITICAL) Fayl İcazə (Permission) Matrisi (KRİTİK)

If the OpenSSH daemon detects that your key files or directories are readable/writable by other users, it will silently ignore the keys and fall back to prompting for a password. This is the #1 cause of SSH key failure. Əgər OpenSSH daemon key fayllarınızın və ya qovluqlarınızın digər istifadəçilər tərəfindən oxuna/yazıla bilən olduğunu aşkarlayarsa, o, səssizcə key-ləri iqnor edəcək və parol tələb etməyə (fall back) keçəcək. Bu SSH key xətalarının 1 nömrəli səbəbidir.

ConfigurationKonfiqurasiya

SSH Daemon Structure

RHEL 9 introduces drop-in directories for cleaner configuration management. Understand file precedence. RHEL 9 daha təmiz konfiqurasiya idarəetməsi üçün drop-in qovluqları təqdim edir. Fayl oxunma üstünlüyünü (precedence) öyrənin.

1. Main Configuration File1. Əsas Konfiqurasiya Faylı

/etc/ssh/sshd_config

The legacy monolithic configuration file. While you can still edit this directly, it is no longer the recommended approach for automation or package upgrades in RHEL 9. Köhnə monolit konfiqurasiya faylı. Hələ də bunu birbaşa redaktə edə bilsəniz də, bu artıq RHEL 9-da avtomatlaşdırma və package yeniləmələri üçün tövsiyə olunan yanaşma deyil.

2. Drop-in Directory (RHEL 9 Standard)2. Drop-in Qovluğu (RHEL 9 Standartı)

/etc/ssh/sshd_config.d/*.conf

Administrators create modular .conf files here (e.g., 99-custom-security.conf). These files are dynamically included by the main config. Administratorlar burada modulyar .conf faylları yaradır (məs. 99-custom-security.conf). Bu fayllar əsas konfiqurasiya tərəfindən dinamik olaraq oxunur.

Parsing Precedence Rule:Oxunma Üstünlüyü (Precedence) Qaydası: The SSH daemon parses directives in order. The FIRST obtained value for a directive is the one used. If PermitRootLogin no is defined in a drop-in file, a later entry in the main sshd_config file will be ignored! SSH daemon direktivləri ardıcıllıqla oxuyur. Bir direktiv üçün əldə edilən İLK dəyər istifadə olunur. Əgər bir drop-in faylında PermitRootLogin no təyin edilibsə, əsas sshd_config faylındakı sonrakı dəyər iqnor ediləcək!

Viewing Active SettingsAktiv Parametrlərə Baxmaq

Because configuration is spread across multiple files, you should always test the configuration and view the effective configuration parsed by the daemon before restarting the service. Konfiqurasiya birdən çox fayla yayıldığı üçün, servisi restart etməzdən əvvəl həmişə sintaksisi test etməli və daemon tərəfindən oxunan effektiv konfiqurasiyaya baxmalısınız.

Test Config Syntax
# Test configuration syntax for errors before reloading [root@server ~]# sshd -t # No output means syntax is valid.
View Effective Output
# Print the combined, active configuration (extended test mode) [root@server ~]# sshd -T | grep -i permitroot permitrootlogin no
SecurityTəhlükəsizlik

Hardening & Custom Ports Hardening və Xüsusi Portlar

Implement strict access controls, disable legacy authentication, and learn the complex process of migrating the default SSH port in an SELinux environment. Ciddi access control tətbiq edin, köhnə autentifikasiyanı ləğv edin və SELinux mühitində default SSH portunu dəyişdirməyin mürəkkəb prosesini öyrənin.

1. Access Control Directives1. Access Control Direktivləri

PermitRootLogin

Controls whether the root user can log in via SSH. Default in RHEL 9 is prohibit-password. Root istifadəçisinin SSH vasitəsilə daxil olub-olmayacağını idarə edir. RHEL 9-da default dəyər prohibit-password-dur.

PermitRootLogin no

Force admins to log in as standard users and use `sudo` or `su`. Admin-ləri standart istifadəçi kimi daxil olmağa və `sudo` və ya `su` istifadə etməyə məcbur edin.

PasswordAuthentication

Allows or denies logging in with an account password. Hesab parolu ilə daxil olmağa icazə verir və ya rədd edir.

PasswordAuthentication no

Apply this ONLY after you have verified Key-Based Auth works. Bunu YALNIZ Key-Based Auth-un işlədiyini yoxladıqdan sonra tətbiq edin.

AllowUsers / DenyUsers

Restrict SSH access to specific users or groups. If AllowUsers is used, ALL other users are implicitly denied. SSH access-ini (girişini) xüsusi istifadəçilərə və ya qruplara məhdudlaşdırın. Əgər AllowUsers istifadə edilərsə, BÜTÜN digər istifadəçilər avtomatik olaraq rədd edilir.

AllowUsers sysadmin dev_ops deploy@10.0.0.*

2. Changing the Default Port (RHCSA Critical) 2. Default Port-un Dəyişdirilməsi (RHCSA üçün Kritik)

Security by obscurity isn't perfect, but changing the port from 22 to 8022 reduces log noise from automated botnets by 99%. In RHEL 9, you cannot simply change the config file; SELinux will actively block the daemon from binding to the new port, and Firewalld will drop external connections. Obscurity (gizlətmə) ilə təhlükəsizlik mükəmməl deyil, lakin port-u 22-dən 8022-yə dəyişdirmək avtomatlaşdırılmış botnet-lərdən gələn log səs-küyünü 99% azaldır. RHEL 9-da konfiqurasiya faylını sadəcə dəyişdirə bilməzsiniz; SELinux daemon-un yeni port-a qoşulmasını (binding) aktiv şəkildə bloklayacaq və Firewall xarici bağlantıları drop edəcək.

1. SSH Config

Create /etc/ssh/sshd_config.d/port.conf and define the new port. /etc/ssh/sshd_config.d/port.conf yaradın və yeni port-u təyin edin.

Port 8022
2. SELinux

Instruct SELinux that port 8022 belongs to the ssh_port_t type context using semanage. semanage istifadə edərək SELinux-a port 8022-nin ssh_port_t type context-inə aid olduğunu bildirin.

# semanage port -a -t ssh_port_t -p tcp 8022
3. Firewall

Open the new port in the firewall and reload the rules. Firewall-da yeni port-u açın və roll-ları reload edin.

# firewall-cmd --add-port=8022/tcp --permanent
# firewall-cmd --reload
4. Restart Service

Because a socket/port is changing, use restart instead of reload to ensure the daemon binds properly to 8022. Socket/port dəyişdiyinə görə daemon-un 8022-yə düzgün qoşulduğundan (bind) əmin olmaq üçün reload əvəzinə restart istifadə edin.

# systemctl restart sshd
UtilitiesAlətlər

Secure Transfers & User Switching Təhlükəsiz Transferlər və İstifadəçi Dəyişdirmə

Utilizing the established encrypted tunnel for file transfers and mastering local privilege escalation. Fayl transferləri üçün qurulmuş şifrəli tuneldən istifadə etmək və Privilege Escalation (imtiyaz artırmağı) öyrənmək.

scp (Secure Copy)

Transfers files directly over the SSH protocol. Uses the exact same authentication (keys/passwords) as standard SSH. Faylları birbaşa SSH protokolu üzərindən transfer edir. Standart SSH ilə tamamilə eyni autentifikasiyadan (key-lər/parollar) istifadə edir.

# Push local file to remote /var/www $ scp index.html admin@web:/var/www/ # Pull remote directory recursively (-r) $ scp -r root@db:/var/log/mariadb/ . # Specify custom port (Capital -P) $ scp -P 8022 data.tar.gz user@host:~

sftp (Secure FTP)

An interactive session for browsing remote directories and transferring files. Excellent when you don't know the exact path beforehand. Uzaq qovluqları (remote directories) gözdən keçirmək və faylları transfer etmək üçün interaktiv sessiya. Dəqiq path-i (yolu) əvvəlcədən bilmədiyiniz zaman əladır.

$ sftp admin@webserver Connected to webserver. sftp> cd /var/www/html sftp> get config.php Fetching /var/www/html/config.php to config.php sftp> put update.zip Uploading update.zip to /var/www/html/ sftp> exit

Defis (-) işarəsini başa düşmək RHCSA-da ciddi şəkildə test edilir.

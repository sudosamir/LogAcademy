Execution

The ssh Client & TOFU

Connecting to hosts, running one-off commands, and resolving Trust On First Use (TOFU) host identification errors.

Core Syntax & Critical Options

Command / Option Real-World Case
ssh user@host Standard interactive login. Defaults to current local user if `user@` is omitted.
ssh user@host [command] Non-interactive execution. Run a command, capture output, and close connection automatically.
ssh root@db-server tail -n 20 /var/log/messages
-p [port_number] Connect to a server listening on a custom port.
ssh -p 2222 admin@firewall.local
-i /path/to/private_key Specify an exact identity file instead of the defaults in ~/.ssh/.
-v, -vv, -vvv Verbose debugging. Mandatory when troubleshooting key rejections or connection drops.

Trust On First Use (TOFU) & Known Hosts

SSH mitigates Man-in-the-Middle attacks by tracking server identity keys. System-wide known hosts are stored in /etc/ssh/ssh_known_hosts, while user-specific host fingerprints are stored in ~/.ssh/known_hosts.

Troubleshooting: Host Key Changed
# Attempting to log into a server that was recently reinstalled [student@workstation ~]$ ssh root@serverb @@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@ @ WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED! @ @@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@ IT IS POSSIBLE THAT SOMEONE IS DOING SOMETHING NASTY! Someone could be eavesdropping on you right now (man-in-the-middle attack)! It is also possible that a host key has just been changed. Offending ED25519 key in /home/student/.ssh/known_hosts:12 Host key verification failed.
# Fix: Remove the old cached fingerprint for serverb [student@workstation ~]$ ssh-keygen -R serverb # Host serverb found: line 12 /home/student/.ssh/known_hosts updated.
Authentication

Key-Based Auth

Passwords are vulnerable to dictionary attacks. Deploying cryptographic key pairs is the enterprise standard for SSH authentication.

1

Generate the Pair

RHEL 9 strongly recommends the ed25519 algorithm over legacy RSA.

[student@ws ~]$ ssh-keygen -t ed25519 Generating public/private ed25519 key pair. Enter file to save key (/home/student/.ssh/id_ed25519): [Enter] Enter passphrase (empty for no passphrase): ****** Your identification has been saved in ...id_ed25519 Your public key has been saved in ...id_ed25519.pub
2

Deploy Public Key

ssh-copy-id securely appends your .pub key to the remote system and strictly sets SELinux contexts and permissions.

[student@ws ~]$ ssh-copy-id student@servera student@servera's password: Number of key(s) added: 1 Now try logging into the machine, with: "ssh 'student@servera'" and check to make sure that only the key(s) you wanted were added.

The File Permission Matrix (CRITICAL)

If the OpenSSH daemon detects that your key files or directories are readable/writable by other users, it will silently ignore the keys and fall back to prompting for a password. This is the #1 cause of SSH key failure.

File / Directory Owner & Context Required Permission Failure Symptom
~/.ssh/ Client AND Server 700 Daemon refuses to read authorized_keys.
id_ed25519 (Private) Client Side Only 600 Client explicitly errors: "Permissions 0644 for key are too open."
id_ed25519.pub (Public) Client Side Only 644 None. Safe to share publicly.
authorized_keys Server Side Only 600 Daemon ignores the file; prompts for password.
Configuration

SSH Daemon Structure

RHEL 9 introduces drop-in directories for cleaner configuration management. Understand file precedence.

1. Main Configuration File

/etc/ssh/sshd_config

The legacy monolithic configuration file. While you can still edit this directly, it is no longer the recommended approach for automation or package upgrades in RHEL 9.

2. Drop-in Directory (RHEL 9 Standard)

/etc/ssh/sshd_config.d/*.conf

Administrators create modular .conf files here (e.g., 99-custom-security.conf). These files are dynamically included by the main config.

Parsing Precedence Rule: The SSH daemon parses directives in order. The FIRST obtained value for a directive is the one used. If PermitRootLogin no is defined in a drop-in file, a later entry in the main sshd_config file will be ignored!

Viewing Active Settings

Because configuration is spread across multiple files, you should always test the configuration and view the effective configuration parsed by the daemon before restarting the service.

Test Config Syntax
# Test configuration syntax for errors before reloading [root@server ~]# sshd -t # No output means syntax is valid.
View Effective Output
# Print the combined, active configuration (extended test mode) [root@server ~]# sshd -T | grep -i permitroot permitrootlogin no
Security

Hardening & Custom Ports

Implement strict access controls, disable legacy authentication, and learn the complex process of migrating the default SSH port in an SELinux environment.

1. Access Control Directives

PermitRootLogin

Controls whether the root user can log in via SSH. Default in RHEL 9 is prohibit-password.

PermitRootLogin no

Force admins to log in as standard users and use `sudo` or `su`.

PasswordAuthentication

Allows or denies logging in with an account password.

PasswordAuthentication no

Apply this ONLY after you have verified Key-Based Auth works.

AllowUsers / DenyUsers

Restrict SSH access to specific users or groups. If AllowUsers is used, ALL other users are implicitly denied.

AllowUsers sysadmin dev_ops deploy@10.0.0.*

2. Changing the Default Port (RHCSA Critical)

Security by obscurity isn't perfect, but changing the port from 22 to 8022 reduces log noise from automated botnets by 99%. In RHEL 9, you cannot simply change the config file; SELinux will actively block the daemon from binding to the new port, and Firewalld will drop external connections.

1. SSH Config

Create /etc/ssh/sshd_config.d/port.conf and define the new port.

Port 8022
2. SELinux

Instruct SELinux that port 8022 belongs to the ssh_port_t type context using semanage.

# semanage port -a -t ssh_port_t -p tcp 8022
3. Firewalld

Open the new port in the firewall and reload the rules.

# firewall-cmd --add-port=8022/tcp --permanent
# firewall-cmd --reload
4. Restart Service

Because a socket/port is changing, use restart instead of reload to ensure the daemon binds properly to 8022.

# systemctl restart sshd
Utilities

Secure Transfers & User Switching

Utilizing the established encrypted tunnel for file transfers and mastering local privilege escalation.

scp (Secure Copy)

Transfers files directly over the SSH protocol. Uses the exact same authentication (keys/passwords) as standard SSH.

# Push local file to remote /var/www $ scp index.html admin@web:/var/www/ # Pull remote directory recursively (-r) $ scp -r root@db:/var/log/mariadb/ . # Specify custom port (Capital -P) $ scp -P 8022 data.tar.gz user@host:~

sftp (Secure FTP)

An interactive session for browsing remote directories and transferring files. Excellent when you don't know the exact path beforehand.

$ sftp admin@webserver Connected to webserver. sftp> cd /var/www/html sftp> get config.php Fetching /var/www/html/config.php to config.php sftp> put update.zip Uploading update.zip to /var/www/html/ sftp> exit

Substitute User: su vs su -

When direct root login via SSH is disabled, administrators log in as a standard user and switch context. Understanding the hyphen is heavily tested in RHCSA.

Command Environment Behavior Directory Behavior Status
su Keeps the original user's variables and $PATH. Root commands (in /sbin) may not execute natively. Stays in current working directory. Not Recommended
su - Simulates full login. Loads root's .bash_profile, proper $PATH, and environment. Changes directory to /root Best Practice
